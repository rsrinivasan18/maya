## Session 8 — Online/Offline Hybrid Routing + Tiered LLM Fallback

### Goals
- MAYA currently always uses Ollama (local). If internet is available, prefer a better online model.
- If offline, fall back to Ollama with no wasted API calls.
- Tiered chain: Sarvam → Claude → OpenAI → Ollama (each tier only tried if key present + previous tier failed).

---

### What We Built

#### 1. `src/maya/agents/connectivity_checker.py` (new file)
- `ConnectivityChecker.is_online()` — TCP probe to `8.8.8.8:53` (Google DNS)
- No DNS resolution needed; direct IP:port connect; zero dependencies (pure `socket`)
- Timeout configurable via `settings.CONNECTIVITY_TIMEOUT` (default 2s)
- Respects `OFFLINE_MODE` setting — if True, always returns False

**Why 8.8.8.8:53?**
Most reliable IP on the internet. Port 53 (DNS) is always open on a working connection.
A raw TCP connect is faster and lighter than an HTTP request — ideal for a per-turn check.

#### 2. `src/maya/agents/llm_router.py` (new file)
```
call_llm_tiered(messages, is_online, fallback_error_prefix) -> (text, provider)
```

Tier chain when online:
- **Tier 1 — Sarvam** (`api.sarvam.ai/v1/chat/completions`, model `sarvam-m`, 24B)
  - Best for Hindi/Hinglish — built specifically for Indian languages
  - Uses `urllib.request` (stdlib) — zero new dependencies
  - Auth: `api-subscription-key` header
- **Tier 2 — Claude** (`anthropic`, model `claude-haiku-4-5-20251001`, max_tokens=300)
  - Lazy import: `import anthropic as _anthropic` inside try block
  - Only tried if `settings.HAS_ANTHROPIC_KEY` is True
- **Tier 3 — OpenAI** (`openai`, model `gpt-4o-mini`, max_tokens=300)
  - Lazy import: `import openai as _openai` inside try block
  - Only tried if `settings.HAS_OPENAI_KEY` is True
- **Tier 4 — Ollama** (always last resort, never fails if `ollama serve` is running)

When offline: skips Tiers 1–3, goes straight to Ollama.

**Returns `(response_text, provider_label)`** where `provider_label` is one of:
`"sarvam"` | `"claude"` | `"openai"` | `"ollama"` | `"error"`

Caller logs it: `[help_response/sarvam]` — visible in `--debug` trace.

## Session 9 — Semantic + Episodic Memory

### Problem

MAYA was showing Srinika her own verbatim question back at her:
> "Last time you asked about: *What is photosynthesis exactly please can you explain slowly*"

This was mechanical and unnatural — like a machine reading back a transcript.

The root cause: `save_memory` stored raw `user_input` in the `topics` table. `greet_response` displayed it literally.

---

### Three Types of Memory (Architecture)

| Type | Description | Where stored | When written | When read |
|------|-------------|-------------|--------------|-----------|
| **Semantic** | 2-4 word topic extracted by LLM | `topics.topic` column | `save_memory` (hot path) | `greet_response` |
| **Episodic** | 1-sentence session summary | `sessions` table | `farewell_response` (background thread) | `load_memory` + `greet_response` |
| **Procedural** | Concepts mastered | (future) | — | — |

---

### What We Built

#### 1. `src/maya/agents/memory_store.py`

**New `topic` column** in `topics` table:
```sql
ALTER TABLE topics ADD COLUMN topic TEXT NOT NULL DEFAULT ''
```
Safe migration: wrapped in `try/except` — if column already exists, silently ignored.

**New `sessions` table:**
```sql
CREATE TABLE IF NOT EXISTS sessions (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER NOT NULL DEFAULT 0,
    summary    TEXT    NOT NULL DEFAULT '',
    timestamp  TEXT    NOT NULL DEFAULT (datetime('now'))
)
```

**Updated `log_turn(message, intent, session_id, topic="")`**
- `topic` is optional — if blank, `get_recent_topics()` falls back to `message`.

**Updated `get_recent_topics(limit=3)`**
- Now returns `topic` if non-empty, else `message` (backward-compatible fallback).
- Example return: `["photosynthesis", "gravity waves", "Newton laws motion"]`
  instead of `["What is photosynthesis?", "How does gravity work?", "..."]`

**New `save_session_summary(session_id, summary)`**
- Inserts one row into `sessions` table.

**New `get_last_session_summary()` → str**
- Returns most recent session summary, or `""` on first-ever session.

**Updated `reset()`**
- Also clears `sessions` table (`DELETE FROM sessions`).

#### 2. `src/maya/graph/hello_world_graph.py`

**`load_memory` node** — loads `get_last_session_summary()`, injects `last_session_summary` into state.

**`save_memory` node** — before `log_turn()`, makes a short LLM call:
```python
topic_messages = [
    {"role": "system", "content": "Extract the main topic in 2-4 words. Reply with ONLY those words."},
    {"role": "user", "content": user_input},
]
extracted, _ = call_llm_tiered(topic_messages, is_online)
topic = extracted.strip()[:50]
store.log_turn(user_input, intent, session_id=session_id, topic=topic)
```
Uses the same tier chain — fast with Sarvam/Claude (~1s), slower with Ollama.
Step log: `[save_memory] → ok (topic: 'photosynthesis')`

**`_summarize_session_background(message_history, is_online, session_id, db_path)`** (new module-level function)
- Builds compact conversation digest (≤ 800 chars)
- Asks LLM: *"Summarize in ONE sentence starting with 'Srinika explored'..."*
- Stores result in `sessions` table via `store.save_session_summary()`
- Silent on any exception (background — must never crash or block)

**`farewell_response` node** — fires `_summarize_session_background` as a daemon thread:
```python
threading.Thread(
    target=_summarize_session_background,
    args=(message_history, is_online, session_id, db_path),
    daemon=True,
).start()
```
MAYA's goodbye message appears instantly. The summary is written while Srinika reads it.

**`greet_response` node** — 3-tier priority:
```
1. last_session_summary available → "Welcome back! Srinika explored gravity and the water cycle."
2. recent_topics available         → "Last time you explored: photosynthesis, gravity."
3. First ever session              → "Hello! I'm MAYA..."
```

#### 3. `src/maya/models/state.py`
Added `last_session_summary: NotRequired[str]` — loaded by `load_memory`, read by `greet_response`.

---

### Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Topic extraction | Hot path (in `save_memory`) | Adds ~1s online / ~5s offline to turn after response shown |
| Session summary | Background thread (daemon) | Must never delay UX; fires after farewell message returned |
| `topic` fallback | `message` if `topic` is blank | Backward-compat; works on DBs created before Session 9 |
| Thread type | `daemon=True` | Auto-killed if process exits before thread finishes |
| Summary prompt | "Start with 'Srinika explored'" | Forces warm, child-friendly phrasing every time |

## Session 10 — Procedural Memory (Mastery Tracking)

### Problem

MAYA remembered *what* Srinika asked about (semantic memory) and *what she explored last session* (episodic memory), but had no notion of *how well she knows* a topic. A topic visited once vs. five times deserved very different teaching approaches.

---

### What We Built

#### 1. `src/maya/agents/memory_store.py`

**New `mastery` table:**
```sql
CREATE TABLE IF NOT EXISTS mastery (
    topic_key  TEXT    PRIMARY KEY,
    display    TEXT    NOT NULL,
    count      INTEGER NOT NULL DEFAULT 1,
    first_seen TEXT    NOT NULL DEFAULT (datetime('now')),
    last_seen  TEXT    NOT NULL DEFAULT (datetime('now'))
)
```
`topic_key` is lowercased for case-insensitive deduplication ("Gravity" and "gravity" are the same concept).
`display` stores the most recent LLM-extracted form (natural case).

**`_mastery_level(count) → str`** — module-level helper:
| Count | Level |
|-------|-------|
| 1x | curious |
| 2x | learning |
| 3–4x | practiced |
| 5+x | expert |

**`update_mastery(topic)`** — SQLite UPSERT (atomically increment without SELECT):
```python
INSERT INTO mastery (topic_key, display) VALUES (?, ?)
ON CONFLICT(topic_key) DO UPDATE SET
    count     = count + 1,
    last_seen = datetime('now'),
    display   = excluded.display
```

**`get_mastery_summary(limit=5) → list[dict]`** — returns `[{topic, count, level}]` ordered by count DESC.

**`reset()`** — also clears `mastery` table now.

#### 2. `src/maya/models/state.py`

- Real-time bidirectional communication via WebSocket

---

### What We Built

#### 1. `src/maya/web/app.py` (new file)

FastAPI application serving the chat UI and handling WebSocket connections.

**REST endpoints:**
- `GET /` — serves `index.html`
- `GET /api/models` — returns available models based on configured API keys (`auto`, `sarvam`, `claude`, `openai`, `ollama`)
- `GET /api/history` — returns recent topics and mastery summary for the sidebar
- `GET /api/health` — health check (`{"status": "ok"}`)

**WebSocket endpoint (`/ws`):**
- One connection = one learning session (`store.start_session()`)
- Client sends: `{"text": "...", "model": "auto", "agent": "auto"}`
- Server sends in order:
  1. `{"type": "connected", "user_name": "Srinika", "session_count": 5}`
  2. `{"type": "thinking"}` — immediately, before LLM call
  3. `{"type": "response", "text": "...", "intent": "...", "language": "...", "steps": [...], "is_online": true}`
- `asyncio.to_thread()` wraps `maya_graph.invoke()` — keeps the event loop free while the blocking LLM call runs

**Key concept — why asyncio.to_thread()?**
`maya_graph.invoke()` is synchronous (blocks until LLM responds). Running it directly inside `async def` would freeze the entire server. `to_thread()` offloads it to a thread-pool worker.

### What We Built

| State | When | Motion |
|-------|------|--------|

#### 2. Voice Output — SpeechSynthesis (`app.js`)

- `speakText(text, lang)` — creates `SpeechSynthesisUtterance`
- Language-aware: `utt.lang = 'hi-IN'` for Hindi/Hinglish, `'en-US'` for English
- Avatar switches to `talking` state during playback (`utt.onstart`)
- Restores `lastDoneState` when speech ends (`utt.onend` / `utt.onerror`)
- Voice toggle (🔊/🔇) in header, preference persisted in `localStorage`
- Cancels in-progress speech when muting

**`lastDoneState` variable** — tracks the correct state to restore after `talking` ends. Without this, the avatar would get stuck in `talking` state forever.

- `.listening` CSS class pulses the mic button red while active

**Full hands-free loop:**
1. Click mic → speak
2. Speech ends → auto-send
3. MAYA responds via TTS → avatar shows `talking`

#### 4. Bug Fixes

**Bug 1 — History corruption (`hello_world_graph.py`)**

When all LLM tiers failed, the error string was stored as an assistant message in `message_history`. On the next turn, Ollama saw the error as conversation context and generated sympathetic responses about "MAYA's thinking process."

Fix:
```python
# Don't store error responses — they corrupt context for next turn
history_update = [] if provider == "error" else [{"role": "assistant", "content": response}]
```
Applied to both `help_response` and `math_tutor_response` nodes.

**Bug 2 — `force_provider` bypassed `is_online` (`llm_router.py`)**

When a model was selected in the UI, the original code skipped the `is_online` check entirely. Offline users who selected Claude/Sarvam would wait 15s for the API timeout before falling back to Ollama.

Fix: check `is_online` first; if offline, go straight to Ollama regardless of `force_provider`:
```python
if force_provider == "ollama" or not is_online:
    tiers = [_TIER_OLLAMA]
else:
    forced = [t for t in all_tiers if t[0] == force_provider]
    tiers = forced + [_TIER_OLLAMA]
```

**Bug 3 — Ollama timeout too short (`llm_router.py`)**

All tiers had a 15s timeout. Ollama runs on CPU — inference on a long math system prompt + mastery context exceeded 15s and timed out, causing even the fallback to fail.

Fix: per-provider timeouts:
```python
_TIMEOUT = {"ollama": 90, "sarvam": 15, "claude": 15, "openai": 15}
```
